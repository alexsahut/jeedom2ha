## Succès et points forts

### Promesse produit tenue
Le diagnostic n'est pas resté un outil de debug brut. Statuts lisibles, raisons explicables, remédiations actionnables — aucun terme technique MQTT ou JSON brut exposé à l'utilisateur. La règle transverse de l'épic ("transformer la complexité technique en diagnostic métier explicable") a été tenue jusqu'au bout.

### Sécurité structurelle de l'export (4.4)
Le pattern allowlist PHP — construction de chaque champ explicitement plutôt que passage transparent de la réponse daemon — crée une barrière architecturale contre les fuites de secrets, pas une vérification a posteriori. Décision réutilisable.

### Fondations propres pour Epic 5
- Pattern d'unpublish propre (4.3) : `eq_ids_supprimes` + cas spécial dépublication sur changement de politique de confiance
- Traçabilité complète normalisée (objet `traceability`, taxonomie fermée des `reason_code`)
- Logique d'explicabilité : distinction supprimé / exclu / indisponible exposée via le diagnostic

### Adaptation en cours d'épic
La story 4.2bis a été créée mid-epic pour combler un gap de traçabilité découvert pendant 4.2. Ce n'était pas un échec — c'est une capacité d'adaptation qui a servi le produit sans dérive de scope.

### Maturité du workflow
La rigueur du pre-flight Git (scripts `start-story-worktree.sh`, `git-preflight.sh`) et la discipline du déploiement terrain via `deploy-to-box.sh` sont pleinement intégrés dans les habitudes de travail.

## Défis et zones d'amélioration

### Comportements Jeedom core sous-documentés (4.3)
Le mécanisme `configKey` sauvegarde automatiquement via le bouton "Sauvegarder" injecté par le core Jeedom — mais PAS via un bouton custom. Cette hypothèse fausse a nécessité 3 itérations de spec sur 4.3 avant d'être corrigée (Arbitrage V3 → pattern `saveFilteringConfig`).

**Impact évité :** si livré avec l'hypothèse initiale, le bouton "Appliquer et Rescanner" aurait appliqué l'ancienne config et produit un faux sentiment de maîtrise pour l'utilisateur.

**Leçon :** Toute hypothèse sur le comportement Jeedom core doit être explicitement posée et sourcée en spec — jamais assumée.

### Bugs révélés uniquement en validation terrain (4.4)
Deux corrections post-recette identifiées uniquement sur box réelle : JS blob anchor pour le téléchargement, et PHP cmd_name dans la pseudonymisation. Non détectables en test Python unitaire.

**Leçon :** La validation terrain n'est pas un bonus optionnel sur les stories touchant le frontend Jeedom et le comportement navigateur. Elle doit être structurellement intégrée dans le workflow, avec runbook et preuves attendues explicites.

### Sous-estimation initiale de l'explicabilité complète
La story 4.2 couvrait cause + remédiation. La story 4.2bis a dû être créée pour couvrir la chaîne de décision complète (objet `traceability` avec `observed_commands`, `typing_trace`, `decision_trace`, `publication_trace`). La nuance entre "exposer une cause" et "exposer la chaîne de décision" n'était pas visible en backlog initial.

## Insights clés

1. **La cohérence de bout en bout est le vrai challenge** — aligner UX ↔ persistance Jeedom ↔ moteur de filtrage ↔ republication MQTT ↔ diagnostic, sans jamais casser la promesse "ne pas mentir à l'utilisateur". Pas la complexité technique unitaire.

2. **Compréhensible, honnête et propre dans le temps** — cette définition de la qualité, posée explicitement en rétrospective, est à porter dans chaque story Epic 5.

3. **Le hardening de spec avant dev prévient les bugs de production** — les 3 itérations de 4.3 sont un coût de spec justifié face au coût qu'aurait représenté un bug de faux sentiment de maîtrise en production.

4. **La validation terrain est un révélateur irremplaçable** — certains écarts (comportement browser, encodage PHP, sérialisation frontend) n'existent que dans l'environnement réel d'une box Jeedom.
