## Défis

### Bug polling Story 2.2 invisible jusqu'au terrain Story 2.3

Le `setInterval` de Story 2.2 vérifiait l'existence de `#div_bridgeStatus` au lieu de `#div_bridgeHealthBanner`. Le polling ne firait jamais. Ce bug n'a été détecté ni par les tests de 2.2, ni par la code review de 2.2, mais par le terrain de Story 2.3 (AC 3 — réactivation ≤ 5s).

**Analyse :** Story 2.2 n'avait pas de test automatisé du comportement dynamique (polling). Le bug était invisible sans test d'intégration JS vérifiant que le polling fire réellement. C'est un signal sur la sécurisation des stories frontend avec comportement dynamique, pas une faute d'implémentation isolée.

**Classification :** dette de process.

### Gap `reasonLabels` porté silencieusement depuis Epic 1

11 `reason_codes` retournés par le backend affichaient "Code inconnu" dans l'UI. Ce gap existait depuis Epic 1, découvert uniquement par l'audit de Story 2.4. Aucun test ne vérifiait que tous les codes backend avaient un label frontend.

**Analyse :** dette de synchronisation de contrat backend/frontend. Le backend peut évoluer et le frontend rester partiellement en retard sans signal fort.

**Classification :** dette technique à traiter comme guardrail pour Epic 3.

### Environnement PHP : dette clarifiée

Le problème n'est pas l'absence de PHP en local (PHP est installé et fonctionne). Ce qui bloque, c'est que certains tests PHP supposent une arborescence Jeedom complète (`core/php/core.inc.php`). La dette réelle est une question d'architecture de tests : classification par profil d'exécution (standalone vs requires-jeedom).

**Classification :** dette de structuration clarifiée en Epic 2, non bloquante rétroactivement. Action item 4 de la rétro Epic 1 reformulé et actionnable.

## Enseignements clés

1. **Le découpage en chaîne de contrats successifs est le bon pattern.** Chaque story consomme le contrat de la précédente sans le modifier. Ce pattern a produit une vélocité et une stabilité nettement supérieures à l'Epic 1.

2. **Vérifier avant de modifier (audit-first) évite la complexité artificielle.** Quand le code existant est déjà sain, le documenter et ne corriger que le delta identifié produit moins de dette et plus de confiance.

3. **Le contrat backend/frontend a besoin d'un garde-fou automatisé.** Les listes statiques de labels dérivent silencieusement. Il faut un test contraignant qui échoue dès qu'un code backend n'a pas de label frontend.

4. **Les comportements dynamiques frontend exigent un filet de sécurité explicite.** Ni les tests unitaires classiques, ni la code review, ni le terrain généraliste ne les attrapent de manière fiable. Le filet (test automatisé ou protocole terrain prescriptif) doit être décidé à la création de la story.

**AI-2 — Remplacement atomique des reason codes existants**
- Owner : Charlie (Dev) + Winston (Architect)
- Scope : quand Story 3.1 remplace les reason codes de fait par la taxonomie formelle, le changement doit être atomique : backend, labels frontend et tests mis à jour dans le même commit/PR.

#### Axe Technique

**AI-3 — Garde-fou de synchronisation contrat backend/frontend**
- Owner : Dana (QA) + Charlie (Dev)
- Scope : créer un test de couverture contraignant qui échoue dès qu'un reason code retourné par le backend n'a pas de label frontend défini. Approche à trancher en 3.1 : fichier source centralisé ou test de découverte dynamique.
- Critère de succès : le test est en place avant que Story 3.2 n'ajoute de nouveaux codes. Un code oublié = test rouge.
- Timing : livré dans Story 3.1 ou au plus tard en début de Story 3.2

**AI-4 — Profils d'exécution des tests PHP : standalone vs requires-jeedom**
- Owner : Charlie (Dev) + Alexandre (infrastructure dev)
- Scope : classifier les tests PHP existants en deux profils. Les tests standalone sont exécutables localement sans socle Jeedom. Les tests requires-jeedom sont marqués avec un skip explicite. Aucun test en zone grise.
- Critère de succès : un agent dev peut lancer les tests PHP standalone et obtenir un résultat fiable.
- Timing : non bloquant pour le démarrage d'Epic 3, déclenchement obligatoire dès qu'une story touche au relai PHP
- Note : dette de structuration clarifiée en Epic 2, non bloquante rétroactivement

#### Axe Tests / Terrain

**AI-5 — Guardrail : filet de sécurité obligatoire pour tout comportement dynamique frontend**
- Owner : Bob (SM) — appliqué à la création de chaque story
- Scope : dès qu'une story introduit de la logique dynamique côté UI (polling, timer, rafraîchissement DOM, état conditionnel), la story doit à sa création spécifier son filet de sécurité : test automatisé JS minimal (pattern `node:test`) ou protocole terrain prescriptif avec timing et état attendu explicites.
- Critère de succès : aucune story avec comportement dynamique frontend ne passe en `ready-for-dev` sans filet de sécurité documenté.
- Timing : applicable dès Story 3.4 (intégration UI)

**AI-6 — Tests d'agrégation cohérents avec les statuts individuels (Story 3.3)**
- Owner : Dana (QA) + Charlie (Dev)
- Scope : les agrégations pièce/global de Story 3.3 sont de la logique métier. Les tests unitaires Python doivent être écrits en même temps que le code, avec des assertions de cohérence entre statuts individuels et agrégats calculés.
- Critère de succès : chaque règle d'agrégation a au moins un test de cohérence. Pas de test ajouté après coup.
- Timing : exécuté dans Story 3.3

#### Axe Process

**AI-7 — Guardrail de synchronisation de contrat intégré aux stories touchant la taxonomie**

- Scope : toute story qui ajoute, renomme ou modifie un reason code ou un statut doit embarquer une vérification de synchronisation backend/frontend dans ses tasks.
- Critère de succès : la code review de chaque story taxonomie vérifie explicitement que le garde-fou AI-3 est respecté.
- Timing : applicable dès Story 3.1

**AI-8 — Capitaliser le pattern audit-first pour les stories de vérification/consolidation**
- Owner : Bob (SM)
- Scope : quand une story a pour objectif de vérifier ou consolider un comportement existant, la stratégie par défaut est audit-first : inspecter, documenter le diagnostic, corriger uniquement le delta.
- Critère de succès : les stories de consolidation documentent "ce qui était déjà sain" avant "ce qui a été corrigé".
- Timing : applicable en continu
