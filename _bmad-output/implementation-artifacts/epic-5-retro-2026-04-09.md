## Ce qui a bien marché

### 1. Backend-first réel, tenu jusqu'au bout

L'invariant architectural numéro un de l'épic — aucune logique métier dans le frontend — a été respecté sur les 7 stories sans exception. Le contrat `actions_ha` backend → UI a joué son rôle d'interface étanche : le JS est resté en lecture pure. Résultat : comportement déterministe, facilité de test, zéro dette côté UI.

### 2. Gate terrain systématique absorbé comme pratique normale

Chaque story a été validée sur la box réelle avant d'être marquée `done` — y compris les micro-stories (5.5, 5.6, 5.7). Le gate terrain a détecté des défauts réels (faux bouton actif → 5.6, compacité insuffisante → 5.5, badge incohérent → 5.7) qui auraient été invisibles en validation locale. Ce n'est plus une contrainte supplémentaire : c'est devenu le mode de fonctionnement normal.

### 3. Micro-stories correctives comme pattern de qualité

Trois micro-stories correctives ont émergé d'observations terrain (post-5.1, post-5.2, post-5.3). Chacune a été traitée avec un scope ultra-ciblé, ses propres critères d'acceptation, et sa propre gate terrain. Ce pattern — constat terrain → micro-story bornée → validation — a permis de lisser l'UX sans casser la dynamique principale et sans introduire de régression.

### 4. Invariants architecturaux stables du premier coup

`_resolve_eq_ids_for_portee` défini en 5.2, réutilisé sans modification en 5.3. Le signal `actions_ha` posé en 5.1, consommé identiquement jusqu'en 5.7. Le snapshot d'opération défini en 5.4, non rouvert. Ces invariants n'ont jamais eu besoin d'être revisités — signal fort de qualité de conception initiale.

### 5. Waiver formel documenté et soldé

Le checkpoint "Créer individuel" non exécuté en gate terrain 5.2 a été formalisé comme waiver explicite, documenté dans la story, et soldé obligatoirement en gate terrain 5.3. Ce n'est pas une dérogation : c'est une pratique de traçabilité qui évite de laisser des hypothèses implicites.

## Ce qui a été difficile

### 1. Absence de modèle système explicite partagé

À plusieurs reprises (notamment entre 5.2, 5.3 et 5.4), il a fallu reconstituer mentalement le comportement global du système : ce qui déclenche une opération, comment le backend calcule le résultat, ce qui est stocké dans `derniere_operation_resultat`, ce que lit le frontend, ce qui est affiché dans le bandeau, et en parallèle ce que fait le gating via `actions_ha`.

Ces informations sont toutes correctes et présentes — dans le code, dans les tests, dans les dev notes — mais jamais externalisées en tant qu'artefact premier partageable. Il ne s'agit pas d'un problème de documentation classique ni de complexité excessive, mais de l'**absence d'un modèle canonique explicite** : un schéma `User Action → Backend Action → Résultat → Stockage → Exposition API → Lecture JS → Affichage UI` avec les invariants, objets clés et transitions nommés.

### 2. Effets plateforme Jeedom implicites

L'auto-sync Jeedom au démarrage du daemon a masqué l'état initial "Aucune" en Story 5.4 (AC4). Ce comportement plateforme, hors périmètre story, était invisible en validation locale et n'est apparu qu'au gate terrain. Il a été géré correctement (validation structurelle via pytest, note terrain documentée), mais il illustre une zone de friction récurrente : des comportements Jeedom implicites qui interfèrent avec la validation terrain sans être recensés comme contraintes explicites.

### 3. Absence de vision produit pendant l'exécution

L'épic a été exécuté avec un focus total sur la livraison — ce qui est correct à court terme. Mais cette intensité d'exécution a laissé peu d'espace pour anticiper ce qui vient après. À la clôture de l'épic, la question "quelle est la suite ?" demande une vraie reprise de hauteur, qui aurait pu être préparée en parallèle.

---

## Cause racine synthétique

Le système d'exécution HA est maîtrisé dans le code. Il n'est pas encore **externalisé en tant que modèle partageable**. Cette asymétrie — code correct, modèle implicite — n'a pas gêné l'exécution de l'Epic 5 mais deviendra un frein dès que le prochain cycle ajoutera de la complexité décisionnelle.

---

## Enseignements clés

1. Le backend-first n'est pas une contrainte : c'est ce qui rend le système testable, déterministe et maintenable.
2. Le gate terrain systématique, appliqué même aux micro-stories, détecte des défauts réels que la validation locale ne voit pas.
3. Une dette nommée et bornée est une décision. Un comportement non maîtrisé est un risque.
4. Un invariant bien conçu n'a pas besoin d'être rouvert. Le soin apporté à 5.1 a payé sur 5.2 à 5.7.
5. **Epic 5 nous a appris à exécuter proprement. Le prochain cycle doit nous apprendre à décider proprement. Et ça ne s'improvise pas.**

---

## Pivot de cycle — Découverte significative

**Le prochain cycle ne peut pas démarrer directement en développement.**

V1.1 Pilotable = *comment exécuter des opérations HA de manière fiable et déterministe.*
Prochain cycle = *comment décider quoi projeter dans Home Assistant.*

Ce n'est pas une évolution de feature : c'est un **changement de paradigme**. D'un système d'actions à un **moteur de projection** — mapping des équipements, typage, règles de décision, overrides utilisateur, explainabilité.

**Risque si non formalisé :** sans modèle canonique du moteur de projection posé en amont, le prochain cycle produira des règles implicites, des overrides difficiles à expliquer et une perte de cohérence progressivement irréversible — exactement le mécanisme qui a justifié l'Epic 4 pour le modèle d'exécution.

L'équivalent de `actions_ha` pour la couche décisionnelle n'existe pas encore. Il doit exister avant que la première story du prochain cycle soit créée.

## Action items

### AI-1 — Flux système opérationnel externalisé

**Owner :** Alexandre + Winston

**Objectif :** externaliser le modèle mental du flux opérationnel sous forme d'un artefact partageable — schéma ou document canonique commenté couvrant la chaîne complète `User Action → Backend → Résultat → Stockage → API → JS → UI`, avec les invariants, objets clés (`actions_ha`, `snapshot`, `derniere_operation_resultat`) et transitions (success / partial / fail).

**Format suggéré :** diagramme Mermaid ou JSON canonique annoté dans `_bmad-output/planning-artifacts/`.

**Critère de succès :** un développeur qui découvre le système peut reconstituer le comportement global sans relire le code.

---

### AI-2 — Effets plateforme Jeedom documentés comme contraintes explicites

**Owner :** Charlie + Dana

**Objectif :** recenser les comportements Jeedom hors périmètre story qui affectent la validation terrain (auto-sync au démarrage, comportements daemon, délais de synchronisation) et les rendre explicites comme contraintes de test documentées — pas comme surprises découvertes en gate terrain.

**Critère de succès :** les stories futures référencent explicitement ces comportements dans leurs dev notes ou leurs critères de validation terrain.

---

### AI-3 — Modèle canonique du moteur de projection (prérequis absolu au prochain cycle)

**Owner :** Alexandre + Winston + Alice

**Objectif :** avant de lancer le prochain cycle, formaliser un modèle canonique du moteur de projection : entrées (équipement Jeedom, type, état, configuration), règles de décision (logique de mapping, ambiguïté, `should_publish`), overrides utilisateur (inclusion/exclusion, priorité), sortie (décision de projection explicable et testable). Ce modèle est l'équivalent de `actions_ha` pour la couche décisionnelle.

**Matière première disponible :** `pub_decision` dans le code (logique existante mais non externalisée).

**Critère de succès :** le modèle est testable indépendamment du code, discutable en revue de planning, et devient le contrat de référence du prochain epic.

**Condition bloquante :** aucune story du prochain cycle ne peut être créée avant que ce modèle soit validé.
